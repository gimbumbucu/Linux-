## 📝 프로젝트 개요 (Overview)
본 문서는 운영체제의 기본 단위인 **프로세스(Process)의 생명주기 및 리소스 할당 메커니즘**을 파악하고, 프로그램 구동 시 발생하는 **메모리(Memory) 구조의 특성**을 심층 분석한 포트폴리오/학습 명세서입니다. 
단순한 코드 작성을 넘어, 시스템의 동작 원리를 이해하여 **메모리 누수(Memory Leak)를 방지**하고 성능을 최적화하는 데 목적이 있습니다.

## 🔬 1. 프로세스 메커니즘 분석 (Process Architecture)

> **[스크린샷 삽입 위치 1: "프로세스란?" 사진을 이곳에 드래그 앤 드롭 하세요]**
> *(여기에 사진을 넣으시면 깃허브가 자동으로 이미지 링크를 변환해 줍니다)*

### 1-1. 프로세스의 정의 및 특징
* **Process (실행 상태의 프로그램):** 하드디스크에 저장된 정적인 코드가 메모리에 로드되어 CPU 리소스를 할당받고 생명력을 얻은 상태를 의미합니다.
* **시스템 자원의 독립성:** 각 프로세스는 독립적인 메모리와 시스템 자원(System Resource)을 할당받아 구동됩니다. 리눅스와 같은 다중 사용자/작업(Multi-tasking) 환경에서는 수많은 프로세스가 동시에 병렬적으로 실행됩니다.
* **계층 구조 식별 메커니즘:** OS는 프로세스를 식별하기 위해 `PID (Process ID)`를 부여하며, 생성 계층 관리를 위해 부모 프로세스인 `PPID (Parent PID)`를 함께 추적합니다.

### 1-2. OS 기반 모니터링: `ps` 명령어
명령줄 인터페이스(CLI) 시스템에서 구동 중인 데몬 및 프로세스를 실시간으로 관제합니다.
* **`ps -e` 옵션:** 사용자 영역 프로세스뿐만 아니라, 시스템 백그라운드 운영에 필수적인 **시스템 프로세스(System Process)** 전체 목록을 스캔하여 리소스 점유 상태를 점검합니다.

## 🧠 2. 프로세스 4대 메모리 구조 (Memory Layout)

> **[스크린샷 삽입 위치 2: "프로그램이란, 프로세스의 특징" 사진을 이곳에 드래그 앤 드롭 하세요]**
> *(여기에 사진을 넣으시면 깃허브가 자동으로 이미지 링크를 변환해 줍니다)*

운영체제는 효율적인 메모리 관리를 위해 각 프로세스에 `Code`, `Static(Data)`, `Stack`, `Heap` 으로 분할된 고유의 메모리 공간을 할당합니다. 이 영역들의 동작 방식을 이해하는 것은 고성능 소프트웨어 설계의 핵심입니다.

### 🧱 1. Stack (스택 영역)
* **특징:** 함수 호출 시 생성되는 **지역 변수(Local Variables)**, 함수의 매개변수, 반환 주소 등 임시 **실행 정보**를 보관합니다.
* **관리:** 컴파일 타임에 크기가 결정되며, 함수의 실행 종료와 함께 (LIFO 방식에 따라) 메모리가 자동으로 해제되는 안전한 영역입니다.

### 🌐 2. Heap (힙 영역)
* **특징:** 애플리케이션 런타임 중 **동적으로 할당된 메모리(Dynamic Memory)**가 적재되는 가장 자유롭고 중요한 공간입니다. (ex: 자바의 `new` 키워드로 생성된 객체)
* **💡 이슈 및 트러블슈팅 포인트:** C, C++ 환경에서 동적 할당 직후 해제(`free`, `delete`)를 누락할 경우 치명적인 **메모리 누수(Memory Leak)**를 유발합니다. Java/Python 환경에서는 가비지 컬렉터(Garbage Collector)가 이를 주기적으로 회수하지만 무분별한 힙 메모리 적재는 GC 오버헤드의 원인이 됩니다.

### 🔒 3. Static / Data (정적 영역)
* **특징:** 전역 변수(Global Variable) 및 `static` 변수가 위치하며, **프로그램이 시작되어 종료될 때까지 메모리에 상주**하는 특수한 구역입니다.
* **생명주기 관리:** 애플리케이션 시작과 동시에 메모리에 **Load**되며, 종료 시 온전히 **소멸**합니다.
* **🚨 최적화 핵심 논점:** 정적 영역은 **가비지 컬렉션(GC)의 수거 대상에서 제외**됩니다. 따라서 `static List<>` 등 전역 컬렉션에 참조 객체를 무한정 삽입하고 비우는 과정을 누락하면 애플리케이션의 메모리를 영구 점유하게 되어 점진적인 성능 저하 및 `Out Of Memory(OOM)` 크래시에 직면할 수 있습니다.

* 네, 포트폴리오에 그대로 '복사-붙여넣기' 하실 수 있도록 **고퀄리티 이미지**와 **전문적인 설명**을 결합한 완성본을 준비했습니다!

## 1. 스레드 (Thread) 개념 이해하기
스레드는 운영체제의 **프로세스 내에서 실행되는 실제 작업의 흐름**을 의미합니다. 여러 개의 작업을 효율적으로 처리하기 위한 핵심 단위로, 다음과 같은 특징을 가집니다.

* **자원 공유와 독립성**: 각 프로세스는 하나 이상의 스레드를 가질 수 있으며, 이들은 같은 프로세스 공간을 공유하면서 동시에 실행될 수 있습니다. 
* **효율적인 메모리 사용**: 여러 스레드는 프로세스 내에서 자원(예: 같은 메모리 공간, 파일 디스크립터 등)을 서로 공유합니다. 그래서 하나의 프로세스 내에서 여러 스레드가 동시에 실행되더라도 시스템 부하를 최소화할 수 있습니다.
* **메인 스레드**: 프로세스는 최소한 하나의 스레드를 생성하여 실행되는데 이를 '메인 스레드'라고 부릅니다. 그 외에 추가적인 스레드를 파생시켜 무거운 연산을 병렬로 처리할 수 있습니다.

### 💡 병렬 작업 (Parallel Processing)이란?
여러 개의 작업을 쪼개서 **동시에 실행하는 것**을 의미합니다. 스레드의 핵심 목적이기도 한 병렬 작업은, 복잡한 연산이나 다중 클라이언트의 요청을 대기 시간 없이 원활하게 처리해 서버의 성능을 극대화시켜 줍니다.

## 2. Bash 쉘 ( `/bin/bash` )
대부분의 리눅스와 유닉스 시스템에서 기본 쉘로 사용되는 Bash (Bourne Again Shell)는, 기존 본 쉘(Bourne Shell)을 확장 및 개선하여 개발된 GNU 프로젝트의 표준 쉘입니다.

### 🔹 값 출력하기 (`echo` 명령어)
* Bash 쉘의 가장 기초적이고 널리 쓰이는 화면 출력 명령어입니다. 지정한 문자열, 텍스트, 쉘 변수 값 등을 터미널 화면에 출력하는 역할을 합니다.
* 일반적으로 터미널 프롬프트에서 시스템 상태를 확인하거나, 쉘 스크립트(또는 배치파일) 구동 중 화면/파일로 상황이나 에러 메시지를 알릴 때 필수적으로 사용됩니다.
* 터미널 환경 변수나 설정 변수의 현재 값을 확인할 경우 주로 사용됩니다. (예: `echo $PATH`)

## 3. 리눅스 표준 입출력 재지정 (Redirection)
리눅스의 가장 강력한 특징 중 하나인 리다이렉션은, 명령어의 결과 출력을 모니터 화면이 아닌 특정 파일로 빼내거나, 사용자의 키보드 입력 대신 파일의 내용을 입력으로 넣어줄 수 있는 기능입니다.

### `>` (표준 출력 재지정, 덮어쓰기)
* 출력 재지정을 사용하면 명령어의 표준 출력 내용을 모니터에 표시하는 대신 특정 파일에 저장할 수 있습니다.
* 만약 이때 지정한 파일이 존재하지 않는다면 자동으로 **새로운 파일이 생성**됩니다. 
* 기존 파일이 존재한다면, **기존 내용은 모두 지워지고 새로운 내용으로 덮어쓰기** 처리됩니다.

### `>>` (표준 출력 추가, 이어쓰기)
* `>` 기호를 사용하면 기존 덮어쓰기를 하므로 기존의 중요한 내용이 날아가는 문제가 생깁니다.
* `>>` 기호를 이용한다면 데이터의 손실 없이 명령어의 출력 결과를 **기존 파일의 맨 끝에 안전하게 이어서 추가**할 수 있습니다. (시스템 로그 기록 등에 널리 활용)

### `<` (표준 입력 재지정)
* 프로그램이 필요로 하는 표준 입력을 사용자의 키보드 타이핑이 아닌, **준비된 파일의 내용으로 대신** 입력합니다.
* 매번 같은 데이터를 반복 입력해야 하거나 대량의 데이터를 스크립트로 일괄 처리할 때 입력 대기 시간을 단축하고 자동화를 구현하는 핵심 기술입니다.

## 🎯 학습 및 적용 결론 (Key Takeaways)
1. **리눅스 환경 완벽 제어:** `ps` 등의 시스템 유틸리티를 활용하여 백엔드 서버 시스템 상에서 돌아가는 데몬(Process)을 안정적으로 모니터링할 수 있는 역량 확보.
2. **코드 레벨의 튜닝:** `Static` 및 `Heap` 등 각 데이터가 어느 메모리 계층에 적재되는지 완벽히 이해함으로써, **메모리 릭 방지** 및 불필요한 GC 개입을 최소화하는 클린 코드를 작성할 수 있는 기반 마련.
